\section{Introduction and Prior Work}

GateGuard is a campus access-control system built for IIT Gandhinagar. It tracks member entry and exit through physical gates, recording each visit in a PostgreSQL database backed by a Node.js REST API and a React frontend.

\subsection{Assignment 1 -- Schema and Seed Data}
We designed the relational schema from the problem specification. Core tables are \texttt{Member}, \texttt{Gate}, \texttt{PersonVisit}, and \texttt{Vehicle}. 5,000 members and over 50,000 person visits were seeded with procedurally generated realistic data.

\subsection{Assignment 2 -- B+ Tree Index and REST API}
A B+ Tree index was implemented from scratch in Python and integrated with a Node.js/Express REST API exposing member management, visit recording, gate management, and vehicle tracking endpoints.

\subsection{Assignment 3 -- ACID Transactions}
A Write-Ahead Log (WAL), transaction manager with LIFO undo stacks, and a consistency checker were implemented. The race condition in the entry-recording endpoint was fixed using \texttt{SELECT FOR UPDATE} inside an explicit PostgreSQL transaction. Concurrent isolation was verified with a multi-threaded notebook test.

\subsection{Assignment 4 -- Horizontal Sharding (this report)}
This assignment adds horizontal sharding to the GateGuard data layer. Rows are distributed across three logical shards using a hash on the member identifier. The same application logic runs unchanged at the HTTP layer; only the data access layer is shard-aware.

\subsection{Selected Key and Hash Function}

\[ \text{shard\_id} = \text{memberid} \;\bmod\; 3 \]

\texttt{memberid} was selected because it satisfies all three criteria for a good shard key: high cardinality, query alignment (every hot-path API call filters by it), and stability (SERIAL integers never change after assignment).

\subsection{Script 01: Creating Shard Tables}

\texttt{LIKE ... INCLUDING ALL} copies column definitions, constraints, and defaults. Foreign keys are excluded by design -- shards are shared-nothing nodes with no cross-shard referential enforcement.

\begin{lstlisting}[style=sqlstyle, caption={01\_create\_shards.sql}]
CREATE TABLE IF NOT EXISTS shard_0_member (LIKE member INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_1_member (LIKE member INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_2_member (LIKE member INCLUDING ALL);

CREATE TABLE IF NOT EXISTS shard_0_personvisit (LIKE personvisit INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_1_personvisit (LIKE personvisit INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_2_personvisit (LIKE personvisit INCLUDING ALL);
\end{lstlisting}

\subsection{Script 03: Verifying Migration Integrity}

\section{SubTask 3: Query Routing Implementation}

\subsection{Python Router (Module\_A)}

\begin{lstlisting}[style=pystyle, caption={shard\_router.py -- core hash functions}]
NUM_SHARDS = 3

def get_shard_id(member_id: int) -> int:
    return member_id % NUM_SHARDS

def member_table(member_id: int) -> str:
    return f"shard_{get_shard_id(member_id)}_member"

def visit_table(member_id: int) -> str:
    return f"shard_{get_shard_id(member_id)}_personvisit"
\end{lstlisting}

\subsubsection{Single-Shard Path -- O(1)}

For \texttt{memberid = 42}: $42 \bmod 3 = 0$, so only \texttt{shard\_0\_member} is queried. Shards 1 and 2 receive zero network traffic.

\begin{lstlisting}[style=pystyle, caption={shard\_router.py -- O(1) point lookup}]
def lookup_member(member_id: int) -> dict:
    shard_id = get_shard_id(member_id)
    table    = member_table(member_id)
    cur      = get_shard_cursor(shard_id)
    cur.execute(f"SELECT * FROM {table} WHERE memberid = %s", (member_id,))
    return cur.fetchone()
\end{lstlisting}

\subsubsection{Scatter-Gather Path}

\begin{lstlisting}[style=pystyle, caption={shard\_router.py -- scatter-gather range query}]
def scatter_gather_visits(start_dt: str, end_dt: str) -> list:
    results = []
    for i, table in enumerate(all_visit_tables()):
        cur = get_shard_cursor(i)
        cur.execute(
            f"SELECT * FROM {table} WHERE entrytime BETWEEN %s AND %s",
            (start_dt, end_dt)
        )
        results.extend(cur.fetchall())
    return sorted(results, key=lambda r: r["entrytime"])
\end{lstlisting}

\subsection{JavaScript Router (Module\_B)}

\begin{lstlisting}[style=jsstyle, caption={shardRouter.js}]
const NUM_SHARDS = parseInt(process.env.NUM_SHARDS || '3', 10);

function getShard(memberId) {
  // parseInt guard -- query params arrive as strings
  return parseInt(memberId, 10) % NUM_SHARDS;
}

function memberTable(memberId) { return `shard_${getShard(memberId)}_member`; }
function visitTable(memberId)  { return `shard_${getShard(memberId)}_personvisit`; }

module.exports = { getShard, memberTable, visitTable, allVisitTables, allMemberTables };
\end{lstlisting}

The \texttt{parseInt} guard is required because Express query parameters arrive as strings. Applying modulo to a string produces \texttt{NaN}, which would silently route every request to \texttt{shard\_NaN\_member}.

The \texttt{SELECT FOR UPDATE} fix from Assignment 3 is unchanged. The only delta is that the open-visit check and the insert target the shard table:

\section{SubTask 4: Scalability Trade-offs Analysis}

\subsection{Horizontal vs. Vertical Scaling}

Vertical scaling upgrades a single server. It is simple but has a hardware ceiling and creates a single point of failure. Horizontal sharding adds servers. Each shard owns a subset of data and can fail independently, providing graceful degradation unavailable in a single-server setup.

\begin{infobox}[title={Graceful degradation example}]
If \texttt{shard\_2} were unavailable, only members with $\text{memberid} \bmod 3 = 2$ would be affected. All other members would continue normally. A single PostgreSQL instance provides no such isolation.
\end{infobox}

\subsection{CAP Theorem Positioning}

Our shared-nothing PostgreSQL setup is a CP system during network partitions. Each shard is independently consistent. Cross-shard operations sacrifice Consistency slightly -- a write committed on shard\_1 may not be visible in a scatter-gather read that began before the write flushed. This is the documented consistency window.

\subsection{Known Trade-offs}

\subsubsection{Resharding Cost}

With 3 shards: $\text{id} \bmod 3$. Adding a 4th: $\text{id} \bmod 4$. Approximately 75\% of all rows must relocate. AWS explicitly identifies this as the principal weakness of hash-mod sharding \cite{aws-sharding}. Consistent hashing reduces movement to $\frac{1}{N+1}$ but adds lookup-table complexity.

\subsubsection{Data Skew}

Member count is balanced (33.33\% per shard). However, visit frequency is not uniform. Members who visit campus frequently generate far more rows on one shard than infrequent members, creating a write hotspot even when counts look balanced. AWS calls this the ``frequency'' dimension of shard skew \cite{aws-sharding}.

\subsubsection{Scatter-Gather Consistency Window}

There is no global transaction coordinator. PingCAP describes the resulting read-your-own-write window \cite{pingcap-sharding}: a write committed on shard\_1 on one thread may not be visible in a scatter-gather read that started on a different thread and is currently reading shard\_0. Real systems use distributed commit protocols (e.g., Google Spanner TrueTime) to eliminate this.

\subsubsection{No Global Secondary Index}

Email-based login requires knowing the shard for a given email. We use scatter-gather (check all three shards) rather than maintaining a Global Secondary Index (GSI). A GSI would require 2-phase commit on every member write to keep it in sync, and introduces stale-index risk on partial failures. For a three-shard system the scatter cost is acceptable.

\subsubsection{Scale Appropriateness}

Our 5,000-member, 50,014-visit dataset does not require sharding. A single PostgreSQL instance handles tens of millions of rows efficiently with proper indexing. The benefit of sharding becomes significant only above approximately 10 million rows or when write throughput saturates a single node. This assignment proves the mechanism works correctly.

\section{Observations and Conclusion}

The hash-mod strategy on \texttt{memberid} produced a near-perfect 33.32\%/33.34\%/33.34\% distribution across 50,014 visits with zero overlap between shards. All three verification checks passed. The \texttt{SELECT FOR UPDATE} race-condition fix from Assignment 3 required only a one-line table-name change to operate correctly in the sharded context, demonstrating that a well-separated service layer is easy to extend.

\textbf{Limitations:}
\begin{itemize}[leftmargin=1.5em]
  \item No distributed commit protocol -- each shard is independently ACID but cross-shard operations have a consistency window.
  \item Resharding to $N+1$ shards requires migrating approximately 75\% of data.
  \item Email-based login always incurs a 3-shard scatter penalty.
  \item The current dataset size does not benefit from sharding in practice.
\end{itemize}

We implemented sharding across all three layers of the GateGuard stack: SQL migration scripts, a Python shard router with connection pooling, and a Node.js service layer with transparent hash-based routing. The proof notebook demonstrates all four subtasks using live data from the PostgreSQL instance.
