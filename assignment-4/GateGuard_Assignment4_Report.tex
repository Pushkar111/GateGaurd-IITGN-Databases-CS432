\section{Introduction}
\label{sec:intro}

\subsection{What GateGuard Is}

GateGuard is our campus gate management system that we built across three assignments in CS432 before this sharding phase. By the end of Assignment 3, we had a complete end-to-end system with member entry and exit workflows, role-based APIs, a frontend dashboard, and transaction-safe behavior under concurrent load.

Assignment 4 changed the scale question. Instead of asking if the logic works on one database, we asked how to keep the same behavior when data is split across multiple database nodes. This report documents exactly how we selected the shard key, partitioned data across three PostgreSQL containers, routed queries through Python and JavaScript routers, and validated correctness with notebook evidence.

\subsection{What Sharding Means in Plain Language}

\begin{tcolorbox}[insightbox]
Sharding means splitting one large table into multiple smaller tables that live on different database nodes. A router decides where each request should go. If we choose a good shard key, most requests hit exactly one shard, which keeps latency and load predictable even as row count grows.
\end{tcolorbox}

\subsection{Hash Function}

\[
\texttt{shard\_id} = \texttt{memberid} \bmod 3

\subsection{Script 01: Creating Shard Tables}

We used LIKE ... INCLUDING ALL to clone structure, defaults, and constraints so each shard table remains schema-compatible with the original source tables. Foreign keys were intentionally not enforced across shard nodes because this architecture is shared-nothing.

\begin{lstlisting}[style=sqlstyle,caption={01 create shards SQL excerpt}]
CREATE TABLE IF NOT EXISTS shard_0_member      (LIKE member      INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_1_member      (LIKE member      INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_2_member      (LIKE member      INCLUDING ALL);

CREATE TABLE IF NOT EXISTS shard_0_personvisit (LIKE personvisit INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_1_personvisit (LIKE personvisit INCLUDING ALL);
CREATE TABLE IF NOT EXISTS shard_2_personvisit (LIKE personvisit INCLUDING ALL);
\end{lstlisting}

def scatter_gather_visits(start_dt: str, end_dt: str) -> list:
    results = []
    for i, table in enumerate(all_visit_tables()):
        cur = get_shard_cursor(i)
        cur.execute(
            f"SELECT * FROM {table} WHERE entrytime BETWEEN %s AND %s",
            (start_dt, end_dt)
        )
        results.extend(cur.fetchall())
    return sorted(results, key=lambda r: r["entrytime"])

\subsection{Known Trade-offs}

\subsubsection{Resharding Cost}
\begin{tcolorbox}[warnbox]
With N = 3 shards, we use id mod 3. If we add a fourth shard, routing becomes id mod 4 and most rows move. For this dataset, that means roughly 37,510 PersonVisit rows and 3,750 Member rows need migration.
\end{tcolorbox}

\[
\text{Records that keep shard: } id \bmod 3 = id \bmod 4
\]
\[
\text{Rows that move} \approx 75\%
\]

\subsubsection{Data Skew Risk}
\begin{tcolorbox}[warnbox]
Row count distribution is balanced, but write frequency per member is not uniform. A few very active members can create heavier write pressure on a specific shard even when member counts look perfectly even.
\end{tcolorbox}

\subsubsection{Scatter-Gather Consistency Window}
\begin{tcolorbox}[warnbox]
There is no global transaction coordinator. If one shard commits between two fan-out reads, the merged result may include or miss that write depending on timing.
\end{tcolorbox}

\subsubsection{No Global Secondary Index}
\begin{tcolorbox}[warnbox]
Email login does not know memberid in advance, so it must query up to three shards. A global secondary index could reduce this cost but introduces distributed write coordination complexity.
\end{tcolorbox}

\subsubsection{Application Router as Single Point of Failure}
\begin{tcolorbox}[warnbox]
Current routing logic is embedded in the application process. If the app process is down, routing is down. Production deployments usually externalize this layer into redundant routing services.
\end{tcolorbox}

\subsubsection{Sharding Not Strictly Needed at Current Size}
\begin{tcolorbox}[insightbox]
At 5,000 members and 50,014 visits, one PostgreSQL instance can handle this workload comfortably. This assignment is about correctness and architectural readiness, not immediate performance necessity.
\end{tcolorbox}

\section{Observations and Limitations}
\label{sec:observations}

\subsection{What Worked Well}

The modulo hash function turned out to be the cleanest part of the system. We implemented it once in Python, once in JavaScript, and got deterministic behavior everywhere. The distribution came out almost perfectly balanced, which gave us confidence early.

The Assignment 3 concurrency fix carried over smoothly. Because routing logic is isolated, we only needed a small table-name substitution in the critical service path while keeping lock and transaction behavior intact. That separation made this upgrade much less risky than we expected.

The proof notebook became our strongest artifact. Every cell has a narrow purpose, every output is visible, and pass-fail outcomes are explicit. Anyone can rerun the notebook and compare outputs directly with the report.

\subsection{Limitations}

\begin{enumerate}[leftmargin=*]
\item No distributed commit protocol exists across shards; cross-shard operations can show a consistency window.
\item Resharding to N+1 shards requires large-scale row movement and we do not implement online resharding in this assignment.
\item Email-based login still incurs scatter-gather overhead without a global secondary index.
\item Application-level router is a single point of failure in current architecture.
\item Dataset size does not require sharding yet; this is primarily a correctness and architecture exercise.
\end{enumerate}
