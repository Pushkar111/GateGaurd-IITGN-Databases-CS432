\section{Introduction}
\label{sec:introduction}

\subsection{Problem Statement}

In campuses, residential complexes, and corporate facilities, entry and exit management is often handled by security personnel using handwritten logs or verbal verification. This manual process is slow, error-prone, and vulnerable to security breaches, as unauthorised visitors or vehicles can enter without proper tracking.

During peak hours, long queues form at gates, causing congestion and increasing the risk of missed or incorrect entries. Additionally, authorities lack real-time visibility into who is on the premises, making it difficult to respond to emergencies or monitor occupancy levels effectively.

\subsection{Proposed Solution}

\textbf{GateGuard} is an Entry Gate Management System designed specifically for the IIT Gandhinagar campus to address these challenges. The system provides:

\begin{itemize}[leftmargin=*, nosep]
    \item \textbf{Automated Entry/Exit Tracking} -- Recording every person and vehicle entry/exit with timestamps and gate identification.
    \item \textbf{Real-time Occupancy Monitoring} -- Maintaining live occupancy counts at each gate for congestion analysis.
    \item \textbf{Role-Based Access Control} -- Implementing a Guard/Admin/SuperAdmin hierarchy for system security.
    \item \textbf{Comprehensive Visitor Tracking} -- Categorising members as Residents, Students, or Visitors with distinct access policies.
    \item \textbf{Vehicle Registration \& Verification} -- Linking vehicles to owners and tracking vehicle movements independently.
    \item \textbf{Multi-Gate Synchronisation} -- Supporting different entry and exit gates for the same visit, reflecting real campus operations.
    \item \textbf{Audit Trail} -- Maintaining \texttt{CreatedAt}/\texttt{UpdatedAt} timestamps on all records for accountability.
\end{itemize}

\subsection{Project Scope}

The GateGuard system is built on a robust relational database architecture comprising \textbf{10 interconnected entities}, organised into four functional layers:

\begin{enumerate}[leftmargin=*, nosep]
    \item \textbf{Member Management} -- \entityname{MemberType}, \entityname{Member}
    \item \textbf{Vehicle Management} -- \entityname{VehicleType}, \entityname{Vehicle}
    \item \textbf{Gate Control} -- \entityname{Gate}, \entityname{GateOccupancy}
    \item \textbf{Security \& Users} -- \entityname{Role}, \entityname{User}
    \item \textbf{Visit Tracking} -- \entityname{PersonVisit}, \entityname{VehicleVisit}
\end{enumerate}

The database follows \textbf{Third Normal Form (3NF)} to ensure data integrity, eliminate redundancy, and maintain consistency. With \textbf{12 foreign key relationships}, \textbf{30+ constraints}, and \textbf{105 sample records}, the system demonstrates a production-ready design.

This Module B report demonstrates:

\end{enumerate}

\subsubsection{Sequence Diagram Description}

The sequence diagram captures two main workflows:

\textbf{Entry Flow (17 actions):}
\begin{enumerate}[leftmargin=*, nosep]
    \item Guard verifies ID card of approaching person.
    \item Guard approves entry and enters Member ID into the terminal.
    \item System checks if person has a vehicle (decision point).
    \item If vehicle present: Guard enters license plate and vehicle type; system finds or creates vehicle record.
    \item If no vehicle: System bypasses vehicle processing.
    \item System executes three \textbf{parallel operations} (FORK):
    \begin{itemize}[nosep]
        \item Path 1: \texttt{INSERT INTO PersonVisit} (entry record)
        \item Path 2: \texttt{INSERT INTO VehicleVisit} (conditional, if vehicle present)
        \item Path 3: \texttt{UPDATE GateOccupancy SET OccupancyCount = OccupancyCount + 1}
    \end{itemize}
    \item System displays confirmation; Guard allows entry.
\end{enumerate}

\textbf{Exit Flow (10 actions):}
\begin{enumerate}[leftmargin=*, nosep]
    \item Guard enters Member ID for exit at exit gate.
    \item System finds active visit: \texttt{SELECT FROM PersonVisit WHERE ExitTime IS NULL}.
    \item System executes three \textbf{parallel operations} (FORK):
    \begin{itemize}[nosep]
        \item Path 1: \texttt{UPDATE PersonVisit SET ExitTime, ExitGateID}
        \item Path 2: \texttt{UPDATE VehicleVisit SET ExitTime, ExitGateID} (conditional)
        \item Path 3: \texttt{UPDATE GateOccupancy SET OccupancyCount = OccupancyCount - 1}
    \end{itemize}
    \item System displays exit confirmation; Guard allows exit.
\end{enumerate}

\subsubsection{Activity Diagram Description}

The activity diagram models two swim-lane processes:

\textbf{Entry Process:}
\begin{itemize}[leftmargin=*, nosep]
    \item Begins with ID card verification by the Guard.
    \item \textbf{Decision Node:} ``Has Vehicle?'' -- splits into vehicle and pedestrian paths.
    \item Vehicle path: Enter plate number $\rightarrow$ Find/Create vehicle in database $\rightarrow$ Decision: ``Plate exists?'' (reuse or create).
    \item \textbf{Merge Node:} All paths converge.
    \item \textbf{Fork Node:} Three parallel database operations execute simultaneously.
    \item \textbf{Join Node:} Synchronises all three operations before displaying confirmation.
\end{itemize}

\textbf{Exit Process:}
\begin{itemize}[leftmargin=*, nosep]
    \item Guard enters Member ID for exit at the exit gate terminal.
    \item System locates the active PersonVisit record (\texttt{WHERE ExitTime IS NULL}).
    \item \textbf{Fork Node:} Three parallel UPDATE operations (PersonVisit, VehicleVisit, GateOccupancy).
    \item \textbf{Join Node:} Synchronises all updates before displaying exit confirmation.
\end{itemize}

\subsubsection{Weak Entity}

\entityname{GateOccupancy} is modelled as a \textbf{weak entity} because:
\begin{itemize}[leftmargin=*, nosep]
    \item It cannot exist independently without its parent \entityname{Gate} entity.
    \item Its primary key (\texttt{GateID}) is simultaneously a foreign key referencing \entityname{Gate}.
    \item It is connected via an \textbf{identifying relationship} (``tracks''), represented by a double diamond.
    \item In the diagram, it is shown with a \textbf{double rectangle} border.
    \item Deletion of a Gate cascades to its GateOccupancy record (\texttt{ON DELETE CASCADE}).
\end{itemize}

\section{UML to ER Transition}
\label{sec:transition}

This section describes the systematic methodology used to convert the UML class diagram into the ER diagram, following the principles from \textit{Elmasri \& Navathe, Chapters 7 and 10}~\cite{elmasri2015}.

\subsection{Classes to Entities Conversion}

Each UML class maps directly to an ER entity. Attributes are preserved, but UML methods (behavioural elements) are \textit{not} represented in the ER model, as ER diagrams focus exclusively on data structure.

\textbf{Key Conversion Rules:}
\begin{enumerate}[leftmargin=*, nosep]
    \item UML class name $\rightarrow$ ER entity name (rectangle).
    \item UML private attributes (\texttt{-}) $\rightarrow$ ER attributes (ellipses).
    \item UML stereotypes (\texttt{<<PK>>}) $\rightarrow$ ER key attributes (underlined ellipses).
    \item UML stereotypes (\texttt{<<FK>>}) $\rightarrow$ ER relationships (diamond connectors).
    \item UML methods (\texttt{+}) $\rightarrow$ \textit{Not represented} in ER (data-only model).
    \item UML constraint annotations $\rightarrow$ ER participation constraints (single/double lines).
\end{enumerate}

\subsection{R1: MemberType \texorpdfstring{$\rightarrow$}{->} Member (categorizes)}

\begin{table}[H]
\centering
\small
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{Property} & \textbf{Value} \\
\midrule
Cardinality & One-to-Many (1:N) \\
Participation & Total on Member side; Partial on MemberType side \\

\subsection{R11: Gate \texorpdfstring{$\rightarrow$}{->} GateOccupancy (tracks) -- Identifying}

\begin{table}[H]
\centering
\small
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{Property} & \textbf{Value} \\
\midrule
Cardinality & One-to-One (1:1) \\
Participation & Total on both sides (Identifying relationship) \\

\textbf{Justification:} Each gate has exactly one occupancy record, and each occupancy record belongs to exactly one gate. This is an \textbf{identifying relationship} where GateOccupancy is a \textbf{weak entity} -- it cannot exist without its parent Gate. The occupancy count provides real-time gate congestion monitoring.

\textbf{Real-World Example:}
\begin{itemize}[leftmargin=*, nosep]
    \item Main Gate (GateID = 1) $\leftrightarrow$ GateOccupancy (GateID = 1, OccupancyCount = 5).
    \item East Gate (GateID = 2) $\leftrightarrow$ GateOccupancy (GateID = 2, OccupancyCount = 3).
    \item West Gate (GateID = 3) $\leftrightarrow$ GateOccupancy (GateID = 3, OccupancyCount = 8).
    \item South Gate (GateID = 4) $\leftrightarrow$ GateOccupancy (GateID = 4, OccupancyCount = 2).
\end{itemize}

\begin{lstlisting}[style=sqlstyle, caption={R11 Implementation}, label={lst:r11}]
CREATE TABLE GateOccupancy (
    GateID INTEGER PRIMARY KEY,  -- PK + FK (identifying)
    OccupancyCount INTEGER NOT NULL DEFAULT 0,
    CONSTRAINT fk_occupancy_gate FOREIGN KEY (GateID)
        REFERENCES Gate(GateID)
        ON DELETE CASCADE  -- Weak entity: cascade on delete
        ON UPDATE CASCADE,
    CONSTRAINT chk_occupancy_count CHECK (OccupancyCount >= 0)
);
\end{lstlisting}

\section{Additional Constraints}
\label{sec:constraints}

Beyond primary key and foreign key constraints, the GateGuard database implements comprehensive integrity constraints to enforce business rules and ensure data validity.

\subsection{CHECK Constraints}

\subsubsection{Temporal Constraints (Visit Tables)}

\begin{lstlisting}[style=sqlstyle, caption={Exit Time Validation}, label={lst:chk-time}]
-- PersonVisit & VehicleVisit
CONSTRAINT chk_exittime_after_entrytime CHECK (
    ExitTime IS NULL OR ExitTime >= EntryTime
)
\end{lstlisting}

\textbf{Purpose:} Ensures logical timing -- a person or vehicle cannot exit before entering.\\
\textbf{Valid:} Entry at 09:00, Exit at 17:00 $\checkmark$ \quad | \quad Entry at 09:00, Exit NULL $\checkmark$ (still on campus)\\
\textbf{Invalid:} Entry at 09:00, Exit at 08:00 $\times$ (rejected by constraint)

\subsubsection{Exit Gate Consistency}

\begin{lstlisting}[style=sqlstyle, caption={Exit Gate/Time Consistency}, label={lst:chk-exit}]
-- PersonVisit & VehicleVisit
CONSTRAINT chk_exitgate_if_exittime CHECK (
    (ExitTime IS NULL AND ExitGateID IS NULL) OR
    (ExitTime IS NOT NULL AND ExitGateID IS NOT NULL)
)
\end{lstlisting}

\textbf{Purpose:} Ensures data consistency -- cannot have an exit time without an exit gate, or vice versa.\\
\textbf{Valid States:} Both NULL (on campus) or both set (exited).\\
\textbf{Invalid:} ExitTime set but ExitGateID NULL, or ExitGateID set but ExitTime NULL.

\subsubsection{Occupancy Count}

\begin{lstlisting}[style=sqlstyle, caption={Occupancy Count Validation}, label={lst:chk-occ}]
CONSTRAINT chk_occupancy_count CHECK (OccupancyCount >= 0)
\end{lstlisting}

\textbf{Purpose:} Occupancy cannot be negative -- minimum is zero (no one at gate).

\subsubsection{Email Format Validation}

\textbf{Valid:} \texttt{arjun.mehta@iitgn.ac.in} $\checkmark$ \quad \textbf{Invalid:} \texttt{arjunmehta} $\times$

\subsubsection{Age Range Validation}

\subsection{Normalization Analysis}

The GateGuard schema satisfies \textbf{Third Normal Form (3NF)}:

\begin{itemize}[leftmargin=*, nosep]
    \item \textbf{1NF:} All attributes are atomic. No repeating groups or multi-valued attributes in the relational schema. Each cell contains a single value.
    \item \textbf{2NF:} All non-key attributes are fully functionally dependent on the entire primary key. No partial dependencies exist (all tables use single-column primary keys).
    \item \textbf{3NF:} No transitive dependencies. Non-key attributes depend only on the primary key, not on other non-key attributes. Type names are stored in separate lookup tables (MemberType, VehicleType, Role) rather than directly in entity tables.
\end{itemize}

\textbf{Design Decisions for Normalization:}
\begin{itemize}[leftmargin=*, nosep]
    \item \entityname{MemberType}, \entityname{VehicleType}, and \entityname{Role} are extracted as separate lookup tables to eliminate transitive dependencies and enable controlled vocabulary enforcement via CHECK constraints.
    \item \entityname{GateOccupancy} is separated from \entityname{Gate} to isolate frequently-updated occupancy counts from relatively static gate information, reducing write contention.
    \item \entityname{PersonVisit} and \entityname{VehicleVisit} are kept as separate tables (rather than a single ``Visit'' table) to avoid NULL-heavy rows and to clearly model the different attributes and constraints of person visits vs.\ vehicle visits.
\end{itemize}

        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_vehicle_owner FOREIGN KEY (OwnerID)
        REFERENCES Member(MemberID)
        ON DELETE SET NULL ON UPDATE CASCADE,
    CONSTRAINT chk_licenseplate_format
        CHECK (LENGTH(LicensePlate) >= 5)
);

-- 5. Gate
CREATE TABLE Gate (
    GateID SERIAL PRIMARY KEY,
    Name VARCHAR(100) NOT NULL,
    Location VARCHAR(255) NOT NULL,
    CreatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UpdatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT chk_gate_name CHECK (LENGTH(Name) > 0)
);

-- 6. GateOccupancy (Weak Entity)
CREATE TABLE GateOccupancy (
    GateID INTEGER PRIMARY KEY,
    OccupancyCount INTEGER NOT NULL DEFAULT 0,
    UpdatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_occupancy_gate FOREIGN KEY (GateID)
        REFERENCES Gate(GateID)
        ON DELETE CASCADE ON UPDATE CASCADE,
    CONSTRAINT chk_occupancy_count
        CHECK (OccupancyCount >= 0)
);

-- 7. Role
CREATE TABLE Role (
    RoleID SERIAL PRIMARY KEY,
    RoleName VARCHAR(50) NOT NULL UNIQUE,
    CreatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UpdatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT chk_role_name
        CHECK (RoleName IN ('Guard', 'Admin', 'SuperAdmin'))
);

-- 8. User
CREATE TABLE "User" (
    UserID SERIAL PRIMARY KEY,
    Username VARCHAR(50) NOT NULL UNIQUE,
    PasswordHash VARCHAR(255) NOT NULL,
    RoleID INTEGER NOT NULL,
    CreatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UpdatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_user_role FOREIGN KEY (RoleID)
        REFERENCES Role(RoleID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT chk_username_length
        CHECK (LENGTH(Username) >= 3),
    CONSTRAINT chk_passwordhash_length
        CHECK (LENGTH(PasswordHash) >= 32)
);

-- 9. PersonVisit
CREATE TABLE PersonVisit (
    VisitID SERIAL PRIMARY KEY,
    PersonID INTEGER NOT NULL,
    EntryGateID INTEGER NOT NULL,
    EntryTime TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ExitGateID INTEGER,
    ExitTime TIMESTAMP,
    VehicleID INTEGER,
    CreatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UpdatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_personvisit_person FOREIGN KEY (PersonID)
        REFERENCES Member(MemberID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_personvisit_entrygate
        FOREIGN KEY (EntryGateID)
        REFERENCES Gate(GateID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_personvisit_exitgate
        FOREIGN KEY (ExitGateID)
        REFERENCES Gate(GateID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_personvisit_vehicle
        FOREIGN KEY (VehicleID)
        REFERENCES Vehicle(VehicleID)
        ON DELETE SET NULL ON UPDATE CASCADE,
    CONSTRAINT chk_exittime_after_entrytime
        CHECK (ExitTime IS NULL OR ExitTime >= EntryTime),
    CONSTRAINT chk_exitgate_if_exittime CHECK (
        (ExitTime IS NULL AND ExitGateID IS NULL) OR
        (ExitTime IS NOT NULL AND ExitGateID IS NOT NULL))
);

-- 10. VehicleVisit
CREATE TABLE VehicleVisit (
    VisitID SERIAL PRIMARY KEY,
    VehicleID INTEGER NOT NULL,
    EntryGateID INTEGER NOT NULL,
    EntryTime TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ExitGateID INTEGER,
    ExitTime TIMESTAMP,
    CreatedAt TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UpdatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_vehiclevisit_vehicle
        FOREIGN KEY (VehicleID)
        REFERENCES Vehicle(VehicleID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_vehiclevisit_entrygate
        FOREIGN KEY (EntryGateID)
        REFERENCES Gate(GateID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_vehiclevisit_exitgate
        FOREIGN KEY (ExitGateID)
        REFERENCES Gate(GateID)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT chk_exittime_after_entrytime
        CHECK (ExitTime IS NULL OR ExitTime >= EntryTime),
    CONSTRAINT chk_exitgate_if_exittime CHECK (
        (ExitTime IS NULL AND ExitGateID IS NULL) OR
        (ExitTime IS NOT NULL AND ExitGateID IS NOT NULL))
);
